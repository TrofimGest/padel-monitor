## Блок 1. Скорость реакции

Хорошие помещения уходят за дни. Сейчас находка ждёт воскресного отчёта —
устраняем этот лаг без удорожания (правила без LLM в ежедневном прогоне).

### 1.1 Мгновенные алерты на эталон — **S**, приоритет №1

**Цель:** узнать о «том самом» помещении в день появления объявления.

**Реализация:**

- `padel_monitor/cli.py::collect_main`: после upsert'а и скоринга нового
  объявления (`state == "new"`) проверять alert-правило:
  - `rule_pass == 1`;
  - `pre_score >= profile.alert_min_score` (дефолт 80);
  - `"unheated" not in flags` (отопление подтверждено или неизвестно);
  - `"far" not in flags` (город или ближний пояс).
- Прошедшие — в одно сообщение Telegram через существующий
  `telegram.send()`: заголовок `🔥 Новый кандидат-эталон`, далее компакт-карточка
  (площадь/высота/отопление/адрес/расстояние/цена/URL + ссылка на карту).
- Антиспам: не больше `alert_max_per_day` (дефолт 3) сообщений в день; лишние
  кандидаты попадут в воскресный отчёт. Счётчик — по таблице `alerts`.
- Новая таблица `alerts(listing_id, ts, kind)` (миграция в `db.MIGRATIONS`) —
  чтобы не алертить повторно и чтобы воскресный судья видел «уже алертили».
- `config.yaml` → `profile.alert_min_score: 80`, `report.alert_max_per_day: 3`.

**Приёмка:** подложить в базу свежий фейковый listing с score 85/near/heated →
`padel-collect` шлёт алерт один раз; повторный прогон — не шлёт.

### 1.2 Алерт на снижение цены — **S**

**Цель:** момент, когда интересный вариант подешевел — лучший момент звонить.

**Реализация:**

- В `collect_main` уже пишутся события `price_change`. После прогона: выбрать
  события за сегодня, у которых listing когда-либо был в `reported` или
  `alerts`, и новая цена ниже старой на ≥ `alert_price_drop_pct` (дефолт 7%).
- Сообщение: `📉 Подешевел вариант из топа: {title}: X → Y BYN (−Z%) {url}`.
- Отметка в `alerts (kind='price_drop')` — не дублировать по тому же событию.

**Приёмка:** вручную изменить price_byn в базе → следующий collect с
изменившейся ценой с сайта не подделать, поэтому тест через фейковое событие
`price_change` + прогон alert-функции юнитом.

## Блок 2. Покрытие и глубина данных

### 2.1 Detail-обогащение топ-кандидатов — **M**, самый большой вклад в точность

**Цель:** судья сейчас видит обрезанное описание из листинга и ≤4 фото.
На detail-страницах — полное описание, все фото, у Realt часто явная высота
потолков в параметрах.

**Реализация:**

- Новый модуль `padel_monitor/enrich.py` + команда `padel-enrich`
  (`[project.scripts]`): берёт кандидатов недели (та же выборка, что
  `padel-candidates`, топ-`enrich_top_n`, дефолт 20), у кого
  `enriched_at IS NULL`, и щадяще (delay 3–5 с) тянет detail-страницы:
  - **Realt**: `pageProps.object` — полное `description`, все `images`,
    параметры (искать высоту: `ceilingHeight` или текст «высота потолков»),
    `contactName`/`agencyName` (телефоны НЕ сохраняем).
  - **Kufar**: `props.initialState.adView.data.initial` — `body` (полный),
    `images`, `ad_parameters` (там бывает высота).
- Обновляет строку listing: `description` (полный, лимит 6000), `images`
  (все), `ceiling_height_m`/`heated`/`area_min_m2`, если нашлись точнее;
  ставит `enriched_at` (миграция-колонка). После обновления — пересчёт
  правил/pre-score этого listing (переиспользовать `_score_listing`).
- Ограничение контракта: `padel-candidates` по-прежнему режет description до
  1500 для судьи, но фото после enrich отдаёт до `vision_max_images_enriched`
  (дефолт 8) для обогащённых.
- В скилл `padel-weekly-report` шаг 2 добавить: `uv run padel-enrich` перед
  `padel-candidates`.
- Стоп-сигналы как в базовых адаптерах (`AdapterStop`) — enrich не должен
  ронять отчёт: ошибки по объявлению пропускаются, отчёт идёт по тому, что есть.

**Приёмка:** после `padel-enrich` у топ-кандидата описание длиннее листингового,
фото больше, у части кандидатов появилась высота; `padel-collect` после этого
не затирает enrich-поля (upsert обновляет только при изменении content_hash —
проверить и при необходимости исключить enrich-поля из перезаписи).

**Риск:** detail-страницы — дополнительная нагрузка на сайты. Митигируется:
только топ-20/нед, только новые (enriched_at), задержки, кеш навсегда.

### 2.2 Новые категории существующих источников — **S**

**Приёмка:** `padel-collect` находит объявления новых категорий; отсев по

### 2.3 Госаукционы аренды (au.nca.by) — **L**, потенциально золотая жила

**Цель:** госимущество сдаётся через аукционы существенно ниже рынка; для
клуба это шанс на большую площадь за полцены. Этого канала нет ни у кого из
«обычных» поисковиков — конкурентное преимущество.

**Реализация (после разведки):**

1. Разведка через browser-harness (скилл `collector-repair` как процесс):
   структура au.nca.by (Нацкадастр, электронные торги аренды), фильтры
   «аренда → Минск/Минский район → нежилые помещения», где лежат данные
   (SSR/API/DOM), пагинация. Результат — `pi/site-specs/auctions.md` +
   доменная заметка в `pi/browser-harness/domain-skills/auctions/`.
2. Адаптер `padel_monitor/adapters/auctions.py` по итогам разведки; контракт
   тот же `Listing` (price = начальная ставка аренды, в `attrs`: дата торгов,
   задаток, срок аренды).
3. Особенность отчёта: у аукционов есть дедлайн — в карточке выделять
   `⏰ торги {дата}`; alert-правило 1.1 распространить на аукционы с
   ослабленным порогом (дедлайны не ждут воскресенья).
4. Флаг `sources.auctions.enabled`, стоп-сигналы стандартные.

**Приёмка:** collect стабильно возвращает лоты Минска; фейковый прогон отчёта
показывает лот с дедлайном.

**Риск:** незнакомая структура сайта, возможна авторизация для деталей.
Скоуп разведки — 1 сессия; если сайт закрыт — задокументировать и отложить.

### 2.4 Включение Megapolis — **S** (на VPS/BY-IP)

Адаптер написан, выключен из-за гео-блока. На машине с BY-IP:
`sources.megapolis.enabled: true` → `uv run padel-collect` → сверить селекторы
по `data/raw/*/megapolis_*.html`, при расхождении — `collector-repair`.
План уже описан в `pi/site-specs/megapolis.md`.

### 2.5 Межсайтовый дедуп — **M**

**Цель:** одно помещение на Realt и Kufar не должно занимать два слота топа
и дважды алертить.

**Реализация:**

- `padel_monitor/dedup.py`: кандидаты в дубли — пары активных объявлений
  разных источников, у которых расстояние координат < 150 м И площадь
  совпадает ±5% (при отсутствии координат — нормализованный адрес:
  улица+дом). Итог — таблица `dup_groups(listing_id, group_id)`.
- Прогонять в конце `collect_main` (дёшево, по активным).
- `padel-candidates`: из группы дублей отдавать одного представителя
  (с бо́льшим числом заполненных полей), в его JSON — `also_on: [url2]`.
- Отчёт: строка «также на kufar: URL».

**Приёмка:** синтетическая пара (одинаковые координаты/площадь, разные
источники) группируется; в выдаче кандидатов один с `also_on`.

## Блок 3. Умнее судейство

### 3.1 Расстояние до метро — **S**

**Реализация:**

- `padel_monitor/geo.py`: статический список станций метро Минска
  (все 3 линии, ~36 станций, координаты) + `nearest_metro(lat, lon) ->
  (name, km)` через тот же гаверсинус.
- `padel-candidates`: каждому кандидату `nearest_metro: {"name":..., "km":...}`.
- `rules.heuristic_score`: бонус `+6` при `km <= 1.5`, `+3` при `<= 3`.
- Карточка отчёта: `Ⓜ Малиновка · 1.2 км` (вместо текущего поля metro,
  которое заполнено только у части объявлений).
- SKILL.md: локация оценивается с учётом метро (клиентопоток без машины).

**Приёмка:** у кандидатов в городе появляется станция и адекватное расстояние.

### 3.2 Карта конкурентов — **S**

**Реализация:**

- `pi/competitors.md` — рукотворный справочник: действующие падел-клубы и
  крупные ракеточные центры Минска (название, адрес, координаты, число
  кортов). Первичное наполнение — разовый ресёрч (5–10 объектов), обновление
  руками.
- `padel_monitor/geo.py::nearest_competitor(lat, lon)` — читает
  YAML-frontmatter/таблицу из файла; `padel-candidates` отдаёт
  `nearest_competitor: {"name":..., "km":...}`.
- SKILL.md, судейство: «ближайший падел в X км» — далеко от конкурентов при
  хорошей локации = недообслуженный район (плюс); вплотную к сильному
  клубу — риск (минус, но не запрет: кластерный эффект тоже бывает).

**Приёмка:** кандидаты несут поле конкурента; судья ссылается на него в why.

### 3.3 Файл предпочтений — **S**, максимум пользы на строчку кода

**Цель:** «обучать» монитор без кода: вкусы, вето, статусы по конкретным
вариантам.

**Реализация:**

- `pi/preferences.md` — свободный текст Трофима. Стартовый шаблон с примерами:
  «не предлагать ТЦ и подвалы ни при каких score», «район Уручье не хочу»,
  «вариант id=175 отпал: дорого», «важнее близость к центру, чем цена».
- SKILL.md, шаг 2: «прочитай `pi/preferences.md`; предпочтения пользователя
  имеют приоритет над общими критериями скилла, но НЕ над жёсткими правилами
  фильтра. Вето-варианты не показывай в топе, укажи одной строкой в конце
  ("скрыто по предпочтениям: N")».
- Файл в git — история изменений вкусов бесплатно.

**Приёмка:** вписать «id=X отпал» → следующий отчёт не показывает X в топе.

### 3.4 Юнит-экономика в вердикте — **S**

**Цель:** «дорого/дёшево» само по себе не отвечает на вопрос «потянет ли
клуб». Даём судье формулу и допущения — получаем % выручки на аренду.

**Реализация:**

- `config.yaml` → новая секция:

  ```yaml
  economics:
    court_hours_per_day: 14      # игровое окно клуба
    utilization: 0.45            # средняя загрузка корта
    price_per_hour_byn: 60       # ставка аренды корта в час
    rent_share_warn: 0.25        # аренда >25% выручки — тревога
  ```

- `padel-candidates`: отдаёт секцию `economics` как есть (данные, не логика).
- SKILL.md, судейство: «посчитай для кандидата: кортов = floor(кусок / 300);
  выручка/мес ≈ корты × hours × 30 × utilization × price; доля аренды =
  аренда_за_кусок / выручка. В `price_note` добавь: "аренда ≈ N% потенциальной
  выручки (при M кортах)". Доля выше rent_share_warn — явный красный флаг в
  risks; ниже 15% — подчеркнуть как сильную сторону».
- Допущения правит Трофим в конфиге по мере изучения рынка.

**Приёмка:** в вердиктах появляется строка с % выручки; при завышенной аренде
кандидат опускается и получает риск.

### 3.5 «Что спросить при звонке» — **S**

**Реализация:**

  `"call_questions": ["...", "...", "..."]` — 2–3 конкретных вопроса,

**Приёмка:** вопросы конкретные и разные у разных кандидатов (не шаблон).

## Блок 4. Удобство пользования

Интерактивный бот (команды в Telegram) сознательно исключён — см. «Не входит».

### 4.1 Ссылки на карту в каждой карточке — **S**

**Цель:** «где это?» — один тап, без копирования адреса.

**Реализация:**

- Координаты уже в базе. В карточке отчёта и в алертах (1.1) — строка-ссылка:
  `🗺 https://yandex.by/maps/?pt={lon},{lat}&z=16` (формат Яндекса: сначала
  долгота). При отсутствии координат — поисковая ссылка по адресу:
  `https://yandex.by/maps/?text={url-encoded адрес}, Минск`.
- Точки внедрения: SKILL.md (структура карточки, шаг 6) + генератор алертов
  в `cli.py` (1.1). Логики почти нет — шаблон строки.

**Приёмка:** ссылка из отчёта открывает карту в правильной точке (выборочно
проверить 3–5 кандидатов, включая вариант без координат).

### 4.2 Watchlist: статусы по вариантам — **M**

**Цель:** вариант из отчёта сейчас показывается один раз и исчезает из виду.
Нужен жизненный цикл: «наблюдаю → позвонил → смотрю вживую → отпало/финалист»,
и чтобы монитор сам напоминал и отслеживал отслеживаемое.

**Реализация:**

- Источник статусов — `pi/preferences.md` (см. 3.3), без отдельного UI:
  Трофим пишет строки вида «watch 175», «позвонил 175: высота 7м, торг
  возможен», «отпало 175: дорого». Судья на каждом отчёте парсит эти пометки.
- Персистентность: новое поле вердикта `"status": "watch|called|visited|
  dropped|finalist"` — судья проставляет его из preferences; сохраняется через
  существующий `padel-save-verdicts` (вердикт пишется целиком в judge_notes).
- `padel-candidates`: новая секция `watchlist` — активные listings, у которых
  в последнем judge_notes `status` ∈ {watch, called, visited, finalist}
  (id, url, кратко, final_score, status, заметка) + их свежие события
  (price_change/gone за неделю).
- Отчёт: постоянный блок `<b>👀 На контроле</b>` — каждый вариант со статусом,
  что изменилось за неделю (цена, исчез — «⚠ снят с публикации, звоните
  выяснять»), и напоминание, если по watch-варианту нет движения 2+ недели.
- Алерты 1.2 (снижение цены) автоматически распространяются на watchlist.

**Приёмка:** пометить в preferences «watch id=X» → следующий отчёт содержит
блок «На контроле» с X; после «отпало id=X» — X уходит из блока; исчезновение
watch-варианта с сайта подсвечивается.

### 4.3 HTML-карта кандидатов недели — **M**

**Цель:** выбирать по географии глазами: все кандидаты недели + конкуренты +
метро на одной карте, а не адреса списком.

**Реализация:**

- Новая команда `padel-map` (`padel_monitor/mapgen.py`): генерирует
  самодостаточный `data/reports/map-YYYY-MM-DD.html` — Leaflet + OSM-тайлы,
  без API-ключей. Слои:
  - кандидаты недели: маркеры с цветом по final_score (зелёный 75+,
    жёлтый 60–74, серый ниже), popup = мини-карточка (score, площадь, высота,
    цена, ссылка на объявление);
  - «На контроле» (4.2) — фиолетовые маркеры;
  - конкуренты из `pi/competitors.md` (3.2) — красные;
  - станции метро (3.1) — точки; окружности 8/13 км от центра — визуализация
    зон near/ok из профиля.
- Данные карта берёт из тех же контрактов: запуск после сохранения вердиктов
  (`padel-map --candidates /tmp/padel-candidates.json --verdicts
  /tmp/padel-verdicts.json`).
- Доставка: `telegram.py` — новая функция `send_document()`
  (`sendDocument` Bot API, multipart); в `padel-telegram` флаг
  `--document <path>`. Скилл шагом 7 отправляет карту файлом вслед за отчётом.
- Ограничение: тайлы OSM грузятся при открытии (нужен интернет) — это ок;
  сам HTML офлайн-самодостаточен по коду (Leaflet inline).

**Приёмка:** файл открывается на телефоне и десктопе, маркеры кликабельны,
ссылки ведут на объявления; в Telegram приходит как документ.

все новые данные проходят через существующие bounded JSON-контракты; после
изменения правил — `uv run padel-rescore`.
