## 1. Зафиксированные требования

| Вопрос | Ответ |
|---|---|
| Где ищем | Минск и Минский район |
| Под что | Падел-корт (точно) |
| Мин. площадь | Выводится из падел-требований, см. §3 |
| Мин. высота потолка | «Любая высокая» → целевая ≥ 6 м, см. §3 |
| Бюджет / цена за м² | Без ограничения — показываем цену, не фильтруем |
| Типы помещений | Любые (склад, ангар, производство, торговое, спорт…), лишь бы подходило |
| Под ремонт | Допустимо |
| Парковка, 1 этаж, отдельный вход, метро/МКАД | Желательно; показывать в отчёте как чеклист |
| Анализ фотографий | Да |
| Отчёт | Telegram, раз в неделю |
| Надёжность | Еженедельно, но гарантированно |
| ПК включён в нужное время | Нет → основная площадка VPS |
| Fallback с домашнего ПК при блокировках | Да |
| Мгновенные уведомления | Не нужны |

## 2. Итоговая архитектура

```text
VPS (основная площадка)
└── cron: ежедневный сбор (1×/день, утро)
    ├── adapters
    │   ├── megapolis-real.by : HTTP → DOM parser
    │   ├── realt.by          : HTTP → __NEXT_DATA__ (pageProps.objects/object)
    │   └── re.kufar.by       : feature flag, cautious SSR __NEXT_DATA__,
    │                           без deep pagination, без телефонов
    ├── normalize → SQLite (listings, events, scores, runs)
    ├── dedup (source+id → canonical URL → fuzzy: адрес+площадь+цена)
    ├── change detection (цена, площадь, описание, исчезновение)
    ├── hard rules: падел-профиль (§3) — только по новым/изменившимся
    ├── LLM text scoring: прошедшие правила (дешёвая модель)
    └── vision photo analysis: top-N кандидатов (высота, колонны, пролёт)
└── cron: еженедельный дайджест (вс, вечер) → Telegram Bot API
└── после каждого успешного прогона: ping healthchecks.io
    (dead-man switch: нет пинга → алерт)

Домашний ПК: резерв для источника, который блокирует VPS-IP
Playwright:  только fallback, если HTTP перестал отдавать данные
browser-harness: только разведка/диагностика/ремонт адаптеров
```

Стек: Python 3.12, `httpx` + `selectolax` (или BeautifulSoup), SQLite,
системный `cron`, Telegram Bot API напрямую через HTTP, один LLM-провайдер
(дешёвая модель для массового скоринга + сильная модель с vision для top-N).
Конфиг — один `config.yaml` с падел-профилем и флагами источников.

## 3. Падел-профиль (правила фильтрации)

Физика падела: корт 20×10 м, с конструкцией ~ 11×21 м ≈ 231 м² на корт;
высота — минимум 6 м без препятствий (рекомендация 8 м); над кортом не должно
быть колонн.

Жёсткие правила (отсеивают до LLM):

- регион: Минск + Минский район;
- площадь ≥ 180 м² (чтобы не терять пограничные объявления с неточной площадью);
- высота: если указана и < 5 м — отказ; 5–6 м — пограничный флаг;
  ≥ 6 м — норма; **не указана — не отсеивать**, пометить `height_unknown`
  и проверять по фото;
- исключения по умолчанию (не подтверждены Трофимом, включены как разумные):
  подвал/цокольный этаж, кабинеты/коворкинги, офисы и торговые точки < 180 м²,
  явные «под офис/салон/магазин» без большого зала.

Скоринговые бонусы (не отсекают): площадь ≥ 450 м² (2+ корта), 1-й этаж
или отдельно стоящее здание, отдельный вход, парковка, близость метро/МКАД,
упомянутая высота ≥ 8 м, «без колонн» / большой пролёт.

Ключевые поля данных первого класса: `ceiling_height_m`, `has_columns`,
`column_note` — извлекаются из текста (regex + LLM), уточняются по фото.

## 4. Vision-стадия (анализ фотографий)

Только для top-N кандидатов недели (например, до 15), прошедших правила и
текстовый скоринг. Задачи vision:

- оценка высоты потолка (по людям, воротам, стеллажам, ферма/балки);
- наличие и шаг колонн — критично: колонны в зоне 20×10 убивают вариант;
- открытый пролёт / форма зала, пригодная под прямоугольник корта;
- состояние (под ремонт допустимо — фиксируем объём работ);
- парковка/вход, если видны.

Результат vision — гипотеза с confidence, не факт; в отчёте помечается «по фото».

## 5. Расписание и надёжность

- **Сбор ежедневный**, отчёт еженедельный. Ежедневный лёгкий обход (1–2 первые
  страницы каждой категории, сортировка по новизне) дешёв, даёт историю цен и
  7 попыток на неделю — одна упавшая ночь не срывает еженедельный отчёт.
- **Отчёт отправляется всегда**, даже пустой: «новых подходящих нет, система
  отработала N/7 дней». Молчание бота = сигнал поломки.
- **Dead-man switch**: после успешного прогона — ping на healthchecks.io
  (бесплатно); пропуск пинга → алерт на email/Telegram.
- Ошибка одного источника не роняет прогон: адаптеры изолированы, статус
  каждого пишется в `runs` и попадает в футер отчёта.
- Стоп-сигналы адаптера: 403/429, маркеры капчи, неожиданно пустой список →
  адаптер останавливается, пишет `needs_manual_check`, остальные работают.

## 6. Формат еженедельного отчёта (Telegram)

Блоки: «Новые за неделю (top-N)», «Изменения цены по ранее найденным»,
«Исчезнувшие из наблюдаемых», футер здоровья системы.

Формат варианта:

```text
[82/100] Склад, 620 м², выс. 7 м (по фото), Минск, ул. …, 4 500 BYN/мес
✓ 1 этаж  ✓ отд. вход  ✓ парковка  ✗ метро далеко  ? МКАД
Почему: пролёт без колонн ~24×30 м, влезают 2 корта.
Риски: высота оценена по фото; отопление не указано.
Фото: колонн не видно, пол бетон, под ремонт.
Источник: realt.by | URL
```

## 7. План внедрения

1. **Неделя 1** — адаптеры Realt + Megapolis, SQLite, dedup, правила,
   текстовый скоринг, Telegram-отчёт. Разработка локально, деплой на VPS
   сразу (ПК обычно выключен — локальная «обкатка 7 дней» из черновика
   отменена, вместо неё обкатка на VPS с ежедневным контролем).
2. **Неделя 2** — vision-стадия, healthchecks, наблюдение за блокировками VPS.
   Если источник блокирует VPS-IP — этот источник переезжает на домашний ПК
   (запуск при включении, anacron-стиль: «не чаще 1 раза в день, при первой
   возможности»), остальное остаётся на VPS.
3. **Неделя 3** — включить Kufar за флагом: только SSR первой страницы
   категорий, без cursor-пагинации, без телефонов; стоп при любых 403/капчах.

## 8. Отличия от черновика (что улучшено и почему)

1. **VPS с первого дня, а не через 7 дней локальной обкатки** — ПК обычно
   выключен, локальная обкатка нереалистична; ПК остаётся резервом.
2. **Ежедневный сбор + еженедельный отчёт** вместо «отчёт 1–2 раза в день»:
   отчёт нужен раз в неделю, но ежедневный сбор даёт надёжность и историю.
3. **Модельный слой упрощён**: Pi/мульти-модельный роутер и эксперименты
   с 8 провайдерами убраны из MVP — один провайдер, две модели
   (дешёвая bulk + сильная vision). Эксперименты — потом, если скоринг слаб.
4. **Vision повышен до полноценной стадии** (в черновике «гипотеза на потом»):
   два решающих падел-критерия — высота и колонны — почти никогда не пишут
   в тексте, фото единственный источник.
5. **`ceiling_height_m` / `has_columns` — поля первого класса** в схеме,
   а не просто часть описания.
6. **Отсутствие высоты в объявлении не отсеивает вариант** — иначе потеряем
   большинство подходящих ангаров/складов.
7. **Пустой отчёт отправляется всегда + dead-man switch** — закрывает
   требование «раз в неделю, но чтобы точно».
8. **Google Sheets и web-dashboard вырезаны из MVP** (были открытым вопросом).
9. **Чеклист желательных атрибутов** (парковка / 1 этаж / отд. вход /
   метро / МКАД) — явный блок ✓/✗/? в каждом варианте отчёта.
